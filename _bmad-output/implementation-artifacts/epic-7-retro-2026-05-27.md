## What Went Well

1. **Pre-epic spike + preparation paid off massively** — Stripe PaymentProvider interface (spike from Epic 6 prep) was production-ready before Epic 7 started. Resend transactional email was added during 7-2 without schema thrashing. The pattern of "spike → interface → implementation" established in Epic 6 worked flawlessly for Stripe.

2. **Review close-out gate enforced successfully** — Implemented from Epic 6 retro A1. Zero stories closed with unresolved findings. Every `[ ]` patch, decision, and defer was addressed before `done`. Compare to Epic 6's 21 unresolved findings in 6-4.

3. **Adversarial pre-dev review prevented scope contamination** — Story 7-3 was blocked mid-refinement (HB1-HB6). 6 hard blockers surfaced before code was written. Result: Story 7-3a split out cleanly. Payment tracking shipped without a half-baked billing engine.

4. **Party Mode multi-agent conversations produced better decisions** — 7-3 refinement used 6 agents simultaneously (Winston, Murat, Mary, Sally, John, Amelia). The AC9a/AC13 split, idempotency schema rewrite, and `invoice.version` optimistic concurrency were products of structured argument, not solo decisions.

5. **Provider abstraction fully scaled** — PaymentProvider interface, TransactionalEmailProvider interface, and Resend/Stripe implementations all followed the registry pattern from Epics 4+6. No direct Stripe SDK imports outside `providers/stripe/`.

6. **Financial safety patterns are now mature** — Integer cents, append-only payments, atomic RPCs, idempotency keys, RLS `::text` cast, audit logging — all applied consistently across 6 stories. No ambiguity about money handling.

7. **Zero production incidents across Stripe webhooks, payment links, email delivery, and financial mutations** — Despite handling payment tokens, credit notes, and concurrent payments.

8. **7-3 review findings were addressed before 7-3a and 7-4 started** — 7-3a's dev agent record explicitly notes closing critical issues pre-implementation. This is the first epic where dependencies genuinely blocked until upstream issues were resolved.

## What Didn't Go Well

1. **Review patches accumulated to ~120 findings** — While the close-out gate enforced resolution, the *volume* of findings per story is higher than Epics 5-6. 7-1 alone had 16 patches. This suggests specification quality varied: some ACs were underspecified (soft dedup mechanics, XSS in email payload, rate limiter without `request.ip`).

2. **Four files exceeded 250-line hard limit in Story 7-3** — `record-payment.ts` (343 lines), `page.tsx` (276), `record-payment-modal.tsx` (266), `packages/types/src/invoice.ts` (277). The original 200-line gate was missed, and even the 250 hard limit was violated. 7-4 dev notes explicitly flagged this but the same compression happened in 7-4 itself (`void-invoice.ts` at 231, `issue-credit-note.ts` at 244).

3. **Time entry picker UI was deferred across 3 stories** — Blocked in 7-1 ("deferred to 7-2"), blocked in 7-2 ("deferred to trigger: Story 7-3"), stripped from 7-3 (HB4 → split to 7-3a), deferred again in 7-3a ("requires design system component; deferred to future UI sprint"). The billing computation is solid but VA users still have no UI to select time entries for invoicing.

4. **`NOT_IMPLEMENTED` runtime rejections create dead paths** — 7-3 AC9a and AC13 intentionally left a `time_entry` source type that works for creation but can't be used through the UI. The path exists in code but is unreachable for users. This is technical debt disguised as scope management.

5. **RLS isolation tests were reactive, not proactive** — Each story added pgTAP tests after implementation. No Story 0 for "test RLS pattern for invoice tables before implementation." Each story repeated the same RLS test patterns slightly differently.

6. **`invoices_status_transition` CHECK constraint remains a no-op** — 7-4 Patch finding: the CHECK is just a status whitelist (`WHERE status IN (all 7 states)`), not a transition guard. Story 7-1 scoped it to `draft → voided`. Subsequent stories never upgraded it. Full state machine validation is app-layer only — DB doesn't enforce transitions.

## Key Insights

1. **Pre-dev adversarial review is the highest-leverage quality tool we have** — 7-3's 6 hard blockers found before code saved more time than any code review finding. The party-mode refinement produced better ACs in 30 minutes than solo drafting did in hours.

2. **File size limits are real and we're hitting them** — Financial stories have dense state machines (status transitions, error codes, edge cases). 200 lines → 250 lines → 300+ lines. The limit exists for readability, not gatekeeping. But when every story hits it, the limit may need recalibration OR we need more aggressive decomposition patterns.

3. **Splitting stories at refinement time works, but deferred UI creates invisible backlogs** — 7-3a was a clean split. But the time entry picker UI is now deferred across 5 story boundaries. At some point, "deferred to next story" becomes "never done." UI sprint needed.

4. **Atomic RPC pattern is the right approach for financial mutations** — 7-1 (invoice creation via RPC), 7-3 (payment via `record_payment_with_concurrency` RPC), 7-4 (void via `void_invoice_and_clear_time_entries` RPC), 7-5 (webhook dedup via `ON CONFLICT`). The RPC-first financial pattern is proven.

5. **Review volume correlates with AC specificity** — Stories with vague ACs (7-1 soft dedup, 7-2 email payload XSS, 7-3 overpayment contract) produced the most findings. Stories with precise ACs (7-4 state machine, 7-5 Stripe decline codes) produced fewer. AC precision is upstream quality.

### Team Agreements

- Deferred UI items are NOT "deferred to next story" — they are deferred to a **defined story** with a ticket number, or they are deferred to backlog with a specific justification
- File size limit: 200 soft / 250 hard. Hitting 250 requires Architect approval or decomposition plan in story file
- Edge case matrix in ACs for financial stories — every story in Epic 7 would have saved ~3 review findings with a pre-written matrix
- Status transition CHECK constraints: either enforce at DB level or remove and document as app-layer. No-op constraints create false confidence.
