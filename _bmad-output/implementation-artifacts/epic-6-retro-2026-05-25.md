## What Went Well

1. **Pre-epic spike made implementation smoother** — `calendar-provider-spike` completed before Epic 6. The `CalendarProvider` interface was production-ready before 6-1 started. Every story after that consumed it without modification. Compare to Epic 4 where EmailProvider was designed mid-story.

2. **Provider abstraction pattern is now mature** — Email (Epic 4), Calendar (Epic 6). Same registry + interface pattern. Proven template for Stripe (Epic 7). Zero Google SDK imports outside `providers/google-calendar/` across all 4 stories.

3. **Incremental consent flow** — VAs with existing Gmail connections got Calendar scopes extended via `include_granted_scopes=true`. No second OAuth dance. Clean UX from reusing Epic 4 patterns.

4. **Adversarial code review caught critical issues** — 63+ findings including cross-workspace data leaks (6-2 H1), phantom DB columns (6-1 C1), non-functional timeout utility (6-2 E-01), wrong signal resolution (6-3 H1). Every one caught before production.

5. **Pattern reuse accelerated delivery** — OAuth PKCE flow, token encryption (AES-256-GCM), agent worker routing, signal writer pattern all reused from Epics 4-5 without modification.

6. **Pre-dev adversarial review became standard** — 6-3 and 6-4 both ran adversarial review before implementation. 6-3's review caught 2 untestable ACs. 6-4's review produced 18 scope decisions. Zero wasted implementation work on either.

7. **Zero production incidents** — Across OAuth, real-time conflict detection, full booking pipeline, saga-based cascade rescheduling, and daily preview generation.

## What Didn't Go Well

1. **6-4 closed with 21 unresolved findings** — Code review identified 7 decision-needed items and 14 patches. Review ran. Findings were documented. Fixes were never applied. Story transitioned to `done` without resolving them. This is a close-out process failure, not a review failure.

2. **Deferred cap gate (A1) not enforced — third epic in a row** — Epic 4 retro created it. Epic 5 retro revised it. Epic 6 still has no enforcement. 6-4 has 7 deferred items with no Architect+PM exception record.

3. **workspace_id filter on service client queries — recurring HIGH finding** — Appeared as H1 in 6-2. Caught again in 6-4 review. Service client bypasses RLS; missing filter = cross-workspace data leak. The learning isn't sticking without lint enforcement.

4. **Zod at boundaries regressed in 6-4** — Fixed in 6-2 (H2), documented as learning in 6-3, then 6-4 review found raw `as` casts across 5 files. Review catches it but the pattern doesn't stick without editor-level enforcement.

5. **Pre-existing test failures noted but never fixed** — `@flow/tokens` (emotional token count), `time-integrity` (anomaly detection), `categorizer` (XML delimiters) mentioned as "not from this story" in all 4 stories. Four stories, same note, zero fixes. Red baseline masks real regressions.

6. **5-1 RLS isolation tests (TD1) still not done** — P0 production risk from Epic 5 retro. Third epic carrying this. Calendar tables (5 new tables) also have zero RLS isolation tests.

7. **AC0 test-first not auditably enforced** — 6-3 notes "implementation + tests in same session." 6-4 has no red commit SHAs. A2 from Epic 5 retro partially applied.

## Key Insights

1. **Pre-epic preparation directly correlates with smooth implementation** — Calendar spike proved it. The spike cost was a fraction of the implementation time it saved. Standard practice from now on.

2. **Code review findings must be resolved, not just identified** — 6-4 is the proof that a thorough review is wasted if the fix-and-verify step is skipped. Close-out gate required.

3. **Gate, not guideline — still the lesson from Epics 4 and 5** — A1 (deferred cap) documented three times, never enforced. Script it or accept it won't happen.

4. **Pre-existing test failures are tech debt that compounds** — Four stories noting the same failures means four stories where real regressions could hide. Fix the baseline.

5. **Review process is strong; close-out process has a gap** — 63+ findings caught, zero incidents. The review earns its place. The gap is between review completion and story `done`.
